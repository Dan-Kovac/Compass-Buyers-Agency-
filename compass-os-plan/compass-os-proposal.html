03 · Ship early. Iterate.Skills in the team's hands, ahead of the OS.

Every capability ships as a Claude skill in Compass's own workspace first. The first batch is in the team's hands inside three weeks. The team uses it, gives feedback, we refine. The proven version gets wired into the OS later. By month four, the dashboard is a tidy version of tools the team already trusts.

First 3 weeks

  • Proposal drafter. Brief notes in, branded proposal out.
  • Brief extractor. Voice note or chat in, structured brief out.
  • Listing capture. Drop a realestate or domain link, get a clean property record back.
  • SEO audit + keyword skill. Page or topic in, audit and content brief out.

Month 2

  • Match-making skill. Brief plus shortlist in, ranked matches out.
  • Compliance summary. Per-client email and chat digest.
  • Newsletter drafter. Property pick in, monthly buyer brief out.

Month 3

  • Website editor / blog drafter. Topic brief in, Sanity-ready draft out.
  • Off-market pitch. Listing details in, agent-to-agent outreach drafted.
  • Onboarding kit drafter. Signed CAF in, welcome pack out.

Month 4 (productised)

  • Reminders engine. Wired into Bruce. Pest + building, settlement, follow-ups.
  • Marketing agents. SEO, newsletter, website editor, all running through the OS.
  • Every earlier skill, now wired into Bruce and the dashboard.

The OS doesn't replace these skills. It connects them to Compass's data, automates the handoffs, and organises everything in one place.

08 · How it gets better over timeThe system improves itself, with Compass in the driver's seat.

Stage 1 is built so the OS gets sharper the more the team uses it. Not magic. Just an architecture that learns from outcomes and a workspace where new agents can be slotted in cleanly when Compass spots a job worth automating.

01
Proposal Writer learns from wins and losses

Every proposal is tagged won, lost, or unanswered. The Proposal Writer reads its own history before drafting the next one. Phrasing, structure and emphasis that converted, get reused. What didn't, gets quietly dropped.

02
Property Matcher learns from agent feedback

Every match Bruce surfaces gets a thumbs up or down from the agent who reviews it. Those signals feed the matching rules. Over weeks, fewer bad matches. Over months, the log starts catching things the team would have missed by hand.

03
New sub-agents slot in cleanly

The architecture is built so a new specialist (a settlement chaser, a vendor enrichment agent, a marketing scheduler) can be added without rebuilding anything. Compass writes a brief, the engineering agent scaffolds it, you test it, you turn it on. Days, not weeks.

04
Claude upgrades inherited automatically

Anthropic releases new Claude models regularly. Because the OS routes through Anthropic directly, every upgrade flows through to your agents the moment Compass switches the model name. No rebuild. No re-engagement. Just better output.

05
The engineering agent keeps shipping

The engineering and frontend agents handed over at M4 are the same ones that built the OS. Compass keeps using them after handover for tweaks, new screens, schema changes. The system stays alive instead of going stale.

06
Quarterly health-checks keep us honest

Four quarterly reviews across the 12 months after handover, included. Async. We look at the system, flag anything drifting, suggest tweaks. Compass decides what to act on. No upsell motion attached.

09 · A note for the longer viewA replicable environment, as a byproduct.

What we're building is a complete environment. The OS, the agents, the proven processes, all in one workspace. Once it's working for Compass, the same environment can be duplicated for a second branch, a JV, or a franchise across the state. New tenant by configuration, not rebuild.

That isn't the purpose of this project. It's a byproduct of building it properly the first time. Worth flagging because the architectural decisions we make in month one are what make it possible later.
