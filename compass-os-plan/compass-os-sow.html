01 · What we're buildingFour outcomes. One system.

Compass OS is the operating system the team runs the business through. Stage 1 is built around four outcomes that compound. Everything in the scope serves at least one of them. Anything that doesn't is out.

01
Legal protection

Per-client compliance file structure, plus an archive-grade backfill of every Compass client since inception. If audited tomorrow, records produce. REBAA, QLD Fair Trading, NSW Fair Trading covered.

02
Conversion velocity

Discovery call to a polished, branded, live-link proposal in under ten minutes. Bruce drafts, the agent iterates in WhatsApp, the client opens a single canonical URL. Higher win rate at $25k a deal.

03
Property log moat

Every off-market property the team hears about gets parsed and filed by Bruce. Bi-directional matching: new brief scans the log, new property scans open briefs. The business's biggest IP, captured.

04
Closed-loop onboarding

Brief signed off, CAF auto-populated and sent via DocuSign, signed, client engaged, property sourcing fires. The ten manual handoffs that exist today collapse to one approval per stage.

03 · How the team uses itThree surfaces. Each picked for the job.

The team doesn't sit at one place to use Compass OS. They use the surface that matches the moment, and the system stitches it together behind the scenes.

Surface 01
Web app

Desktop-first. Sales pipeline, property log browser, dashboards, per-client compliance files. An embedded agent panel pipes the team straight into the agent space without leaving the page.

Surface 02
WhatsApp (Bruce)

Phone-first. Where capture happens. Voice notes, photos, links, "just spoke to X", dropped naturally. Bruce parses, structures, files. In team DMs by default. Silent observer in client groups when added.

Surface 03
Email

Bruce's outbound channel. Live-link proposals, daily briefings (when the data's rich enough), forwarded client comms ingested back into the OS via a Bruce inbox.

04 · The build, in three streamsThree workstreams in parallel. Convergence in the middle.

The web app, Bruce on WhatsApp, and the agent space build in parallel. Each stream stands up on its own. Then they wire together through the convergence phase. Then we polish, test, train, and hand over.

Stream A
The web app

React + Vite + TypeScript on Vercel. Supabase Sydney for data, auth, storage. Built mobile-responsive from the start. The Compass design language, cleanly applied.

  • Login, tenant + user model, RLS scoped to tenant from Day 1
  • Sales pipeline view (the four-phase flow above, made real)
  • Client records, briefs, per-client document library
  • Property log browser with search and filter
  • Dashboards v1: pipeline, conversion, velocity
  • Embedded agent panel that pipes into the agent space
  • Compliance export per client, one click
Quick note on the plumbing. Repo = GitHub, where the code lives. Supabase = the Sydney-hosted database for clients, briefs, properties, files. Vercel = what serves the web app to browsers. All three accounts in Compass's name from Day 1.
Stream B
Bruce on WhatsApp

Bruce as its own entity, with its own WhatsApp business number. Active in team DMs. Silent observer in client groups when the team adds him. WhatsApp is the transport, the OS is the source of truth.

  • Bruce server provisioned and paired to a Compass number
  • DM-active workflow: voice notes, photos, links, natural language
  • Persona, tone, capture rules locked (no chatty, no fluff)
  • Mirror every message into the OS as an append-only event log
  • Auto-file photos and forwarded emails to the right client
  • Operational runbook: rate-limit, heartbeat, re-pair on session drop
Stream C
The agent space

Compass's Anthropic workspace, built like a proper environment. Bruce as orchestrator. Six specialised sub-agents doing the work. Plus the engineering and frontend agents that build the OS.

  • Anthropic workspace owned and billed by Compass
  • Bruce orchestrator: routing, DB writes, persona-consistent replies
  • Brief Extractor: meeting context to structured brief
  • Proposal Writer + Proposal Editor: brief to live-link HTML
  • Property Parser: realestate / domain link to structured property
  • Property Matcher: rule-based brief ↔ property matching
  • Compliance Filer: forwarded artefacts to right client + tag
  • Engineering + frontend agents that build the OS, handed over with prompts and runbooks
Quick note on the workspace. Anthropic workspace = Compass's own login at Anthropic (the company behind Claude), billed and controlled by Compass. The orchestrator is Bruce's brain. Sub-agents are the specialists he hands work to.

08 · How we de-risk deliveryThe three things that could slow this down. Named first.

No build is risk-free. Naming the risks honestly is how we keep them small. Here are the three real ones, and how we handle each.

Risk 01
WhatsApp transport reliability

The whole capture story leans on Bruce's WhatsApp number working reliably. Sessions can drop. Numbers can get rate-limited if outbound is too aggressive. Most of this is solved by treating WhatsApp as a transport, not a database.

How we handle it. WhatsApp is the transport, not the source of truth. Every message mirrors into the OS instantly as an append-only log. If Bruce drops, capture pauses, records survive. We rate-limit outbound, run heartbeat monitoring, and document a re-pair runbook from Day 1.
Risk 02
Historical archive is messier than expected

Two and a half years of WhatsApp + email + Drive folders is a lot of data, and real Compass data is unique. Ingestion may sit lower than expected on first pass.

How we handle it. Stage 1 historical scope is archive-grade only. Files filed, searchable, retrievable, defensible if audited. Structured parsing is a Phase 2 SOW. The acceptance gate is "ingested + searchable", not "100% structured".
Risk 03
Scope grows mid-build

The team will see the build and have ideas. Real client work will surface patterns the original spec didn't catch. Saying yes to everything blows the timeline. Saying no to everything stops Compass shaping its own system.

How we handle it. Anything inside the existing scope gets reordered in real time, no paperwork. Anything outside scope gets a one-page change note: impact, cost, knock-on. Compass decides in writing. No surprises either way.

11 · How it gets better over timeThe system improves itself, with Compass in the driver's seat.

Stage 1 is built so the OS gets sharper the more the team uses it. Not magic. Just an architecture that learns from outcomes and a workspace where new agents can be slotted in cleanly when Compass spots a job worth automating.

01
Proposal Writer learns from wins and losses

Every proposal is tagged won, lost, or unanswered. The Proposal Writer reads its own history before drafting the next one. Phrasing, structure and emphasis that converted, get reused. What didn't, gets quietly dropped.

02
Property Matcher learns from agent feedback

Every match Bruce surfaces gets a thumbs up or down from the agent who reviews it. Those signals feed the matching rules. Over weeks, fewer bad matches. Over months, the log starts catching things the team would have missed by hand.

03
New sub-agents slot in cleanly

The architecture is built so a new specialist (a settlement chaser, a vendor enrichment agent, a marketing scheduler) can be added without rebuilding anything. Compass writes a brief, the engineering agent scaffolds it, you test it, you turn it on. Days, not weeks.

04
Claude upgrades inherited automatically

Anthropic releases new Claude models regularly. Because the OS routes through Anthropic directly, every upgrade flows through to your agents the moment Compass switches the model name. No rebuild. No re-engagement. Just better output.

05
The engineering agent keeps shipping

The engineering and frontend agents handed over at M3 are the same ones that built the OS. Compass keeps using them after handover for tweaks, new screens, schema changes. The system stays alive instead of going stale.

06
Quarterly health-checks keep us honest

Four quarterly reviews across the 12 months after handover, included. Async. We look at the system, flag anything drifting, suggest tweaks. Compass decides what to act on. No upsell motion attached.
